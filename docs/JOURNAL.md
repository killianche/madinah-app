## Особенности, обнаруженные при анализе

### 1. 1 284 строки без `Ученик_id`

Это ≈3.3% от общего числа уроков. Причины:
- в ранних версиях бот не генерировал UUID ученикам при быстрой записи
- импорты из Google Sheets / ручные правки теряли ID

**Как обрабатываем:**
1. Для каждой строки без ID ищем по имени среди уже известных учеников (lowercase match).
2. Если имя известно — привязываем урок к существующему UUID.
3. Если нет — создаём синтетический UUID (в итоге получилось 63 таких ученика).

Менеджер после запуска пройдётся по спискам и при необходимости сольёт дубли (кнопка "Объединить ученика" в админке).

### 2. Два разных "Зейд" в архиве

В журнале есть:
- `Зейд + 20 593 87 22` (последний урок 26.06.2025)
- `Зейд 7225` (последний урок 05.08.2025)

Оба **архивные** (не в активном списке). Новый активный "Зейд" будет заведён через админку отдельно, когда начнёт вести уроки.

### 3. "Шамиль" и "Шамиль Кутбудинов" — разные

В журнале присутствуют:
- `Шамиль` (активный)
- `Шамиль Кутбудинов` (архивный)

Владелец подтвердил: в активном списке "Шамиль" — это не Инаев и не Кутбудинов, а отдельный человек.

### 4. Имена учеников часто содержат телефон

Пример: `Шероз 89162070600`, `Саян 87079883767`. Это не ошибка — в боте имя и телефон хранились одной строкой. В новой системе:
- при импорте парсим по regex — если на конце 10–12 цифр, это телефон → отделяем в колонку `phone`
- остаётся чистое имя

## Что импорт НЕ делает

- **Не создаёт пользователей (`users`).** Записи в `users` создаёт админ отдельно при выдаче доступа конкретному человеку.
- **Не привязывает ученика к учителю.** В журнале ученик мог попадать разным учителям — нет однозначного "текущего учителя". Менеджеры после запуска проставят привязки в админке (или учитель сам добавит себе учеников).
- **Не импортирует пополнения баланса.** Их в CSV нет. Стартовый баланс у всех учеников = 0, менеджеры проставят по каждому после запуска.
- **Не импортирует флаг благотворительности.** Все попадают с `is_charity=false`, менеджеры отметят вручную.
