# Ключевые решения (ADR)

Документ фиксирует **почему** выбраны те или иные инструменты и подходы. Не обязательно читать для работы, но полезно при возвращении к проекту через полгода или при передаче другому разработчику.

---

## ADR-01: Next.js 15 вместо Vue / SvelteKit / чистого React

**Контекст:** нужен мобильный web, быстрый рендер на слабых андроидах (бо́льшая часть учителей), SEO не важен, но SSR полезен для скорости первого экрана.

**Решение:** Next.js 15 (App Router) + React 19.

**Почему:**
- Единая экосистема: фронт и бэк в одном проекте (API routes / Server Actions) — не нужен отдельный Express / Fastify.
- App Router + Server Components: минимум JS на клиенте, быстро на дешёвых телефонах.
- React 19 + использование Server Actions — снимает много бойлерплейта на формы (а у нас всё — формы).
- Огромная экосистема готовых компонентов (shadcn/ui).

**Почему не другое:**
- **SvelteKit** — меньше комьюнити, меньше готовых компонентов. Проигрывает по скорости разработки.
- **Vue / Nuxt** — хороший, но у Next больше inertia в 2026.
- **Pure React + Vite** — быстрее стартует, но придётся самостоятельно ручить роутинг, SSR, код-сплиттинг.

## ADR-02: PostgreSQL 16 самохостед (через managed) вместо Supabase

**Контекст:** нужно соблюсти 152-ФЗ — персональные данные россиян должны храниться на территории РФ. Supabase — это Postgres + auth + RLS + storage — идеально технически, но серверы в ЕС/США.

**Решение:** managed PostgreSQL 16 на TimeWeb Cloud (датацентр Москва) + свой код для auth (Lucia) + своё RLS руками.

**Почему:**
- 152-ФЗ требует локализации ПД в РФ.
- Supabase Ru-версии нет.
- Managed Postgres даёт автобэкапы и мониторинг — отлично.
- Проигрываем в удобстве (без Supabase Studio, без готового auth-UI), но оно того стоит.

**Что теряем:**
- Realtime subscriptions из коробки — для нашего кейса не нужны (журнал не нуждается в live).
- Storage для файлов — в MVP не нужен.
- Auto-generated REST API — справимся написав тонкие серверные функции.

## ADR-03: Свои сессии вместо Lucia / NextAuth / Auth.js / Clerk ⚠ ИЗМЕНЕНО

**Контекст (изначальный):** нужна простая session-based аутентификация. Планировали Lucia v3.

**Что случилось:** перед запуском выяснилось, что **Lucia архивирован автором** (объявил EOL в 2025), а следующий за ним популярный Arctic — только OAuth. Держать основой прод-приложения заброшенный пакет не хочется.

**Новое решение:** свой очень тонкий auth-слой в `web/src/lib/auth/session.ts` (~100 строк):
- токен в cookie = 32 случайных байта, hex-энкодинг
- `id` в таблице `user_sessions` = SHA-256 хэш токена (утечка БД → токены бесполезны)
- TTL 30 дней, `expires_at` проверяется в `getSession()`
- argon2id OWASP parameters (memoryCost: 19_456, timeCost: 2, parallelism: 1)
- `requireAuth()` / `requireRole('teacher', 'manager')` редиректит на `/login` если не авторизован

**Почему не альтернативы:**
- **Clerk / Kinde** — серверы не в РФ, 152-ФЗ.
- **NextAuth/Auth.js** — мощный, но для нашего кейса избыточен.
- **Lucia** — архивирован.
- Своя реализация вмещается на экран и читабельна, это не rocket science когда не нужны OAuth-провайдеры.

## ADR-04: Таблица `teachers` отделена от `users`

**Контекст:** есть учителя, которые больше не работают (38 штук), но их уроки должны оставаться в истории. Если бы был просто `users`, пришлось бы либо оставлять `is_active=false` и не давать логиниться, либо удалять и терять ссылочную целостность у `lessons.teacher_id`.

**Решение:** 
- `teachers` — "кто когда-либо вёл уроки" (active + archived), самостоятельная сущность с UUID
- `users` — "кто может логиниться" (подмножество active teachers + другие роли), ссылается на `teachers.user_id`
- У archived учителей `user_id IS NULL`, ФИО денормализовано в `teachers.full_name`

**Плюс:** уроки всегда указывают на корректного учителя, даже если тот ушёл.
**Минус:** при активации учителя нужно и `teachers.status = 'active'`, и создать `users`.

## ADR-05: Баланс считается в приложении, НЕ триггером

**Контекст:** у каждого ученика есть поле `balance` — сколько уроков оплачено. При каждом проведённом/штрафном уроке -1, при пополнении +N.

**Варианты:**
1. **Триггер в БД** — при insert/update/delete в `lessons` или `balance_topups` автоматически пересчитывать `students.balance`.
2. **Явно в коде** — приложение в рамках транзакции считает и UPDATE-ит.

**Решение:** вариант 2 (явно в коде).

**Почему:**
- Триггеры сложно дебажить (упало — непонятно где).
- Race conditions при параллельных инсертах — триггер без `SELECT ... FOR UPDATE` может дать неверный результат.
- В коде видно что происходит, можно логировать, audit писать.
- Плюс: есть `v_student_full.balance_calculated` — пересчёт на лету из истории. Скрипт `reconcile_balances.sql` еженочно сверяет `students.balance` с `balance_calculated` и алертит при расхождении.

## ADR-06: `audit_log` партиционированный

**Контекст:** каждое действие (создание ученика, запись урока, смена баланса) пишет строку в `audit_log`. За год при 100 уроков/день будет ~36 500 строк — терпимо, но за 3 года уже 110К+. Плюс JSONB с диффом — ещё и жирно.

**Решение:** партиционирование по месяцу (`partition by range (created_at)`).

**Плюсы:**
- Быстрые запросы "что было за последний месяц" — сканируется одна партиция.
- Дроп старых данных = DROP TABLE audit_log_2024_01 — мгновенно, без VACUUM.
- Ретеншен 3 года — раз в месяц автоматически дропаем партицию 37-месячной давности.

## ADR-07: Soft delete для уроков, hard delete для пользователей

**Контекст:** учитель может "удалить" запись урока (исправление ошибки). Но это чувствительная операция — важно иметь возможность аудита.

**Решение:**
- `lessons.deleted_at` — soft delete, в UI не показываем, но физически запись остаётся.
- `students.status = 'archived'` — тоже soft, никогда не удаляем физически.
- `users` — удаляем физически при увольнении сотрудника, но `teachers` остаётся в `archived` (см. ADR-04).

## ADR-08: Claude / Anthropic дизайн-система

**Контекст:** владелец увидел дизайн Claude и попросил сделать "как у него". Дизайн эмоционально-тёплый, непохожий на стандартные CRM/SaaS, хорошо работает на "человеческих" продуктах (школа — именно такой).

**Решение:** палитра Parchment #f5f4ed (фон), Ivory #faf9f5 (карточки), Terracotta #c96442 (акцент), Anthropic Near Black #141413, Olive Gray #5e5d59. Типографика: Anthropic Serif 500 для заголовков, системный sans для UI.

**Технически:**
- Tailwind с кастомными токенами.
- shadcn/ui как база, темизируем через CSS-переменные.
- Ring shadows для карточек (0 0 0 1px #ebe8df) — не box-shadow.
- Закруглённости 8-32px.

## ADR-09: UUID из журнала сохраняем для учителей и учеников, НЕ сохраняем для уроков

**Контекст:** в CSV-журнале у каждой строки есть `Учитель_id` и `Ученик_id` (UUID). У уроков отдельного ID нет — только композитный (дата + учитель + ученик + статус), который не уникален (двухчасовые уроки дают дубликаты).

**Решение:**
- `teachers.id` = UUID из журнала → 39 475 ссылок в `lessons` работают без мэппинга.
- `students.id` = UUID из журнала для тех, у кого он был (кроме 63 синтетических).
- `lessons.id` = новый `gen_random_uuid()` — чтобы не было коллизий.
- `balance_topups.id`, `users.id`, `audit_log` — все новые UUIDv4.

**Плюс:** импорт идёт простой INSERT без SELECT'ов и маппингов → быстрее и надёжнее.

## ADR-10: Нет self-registration

**Контекст:** некоторые бы подумали — пусть учителя сами регистрируются по Telegram-ID или email.

**Решение:** только админ создаёт аккаунты. Причины:
- Школа закрытая, всех знаем по именам.
- Спам / левые регистрации исключены.
- Пароли выдаются лично/в закрытом канале.
- При создании генерируется одноразовый токен для смены пароля при первом входе.

## ADR-11: Метрика "проблемные ученики" — композитный score

**Контекст:** менеджер должен видеть, кто из учеников "в группе риска" (скоро уйдёт / перестанет платить).

**Варианты метрики:**
- Просто "нет уроков 30+ дней" — грубо.
- Композит: дни-с-последнего-урока × 0.3 + отмен-за-месяц × 0.5 + баланс<5 × 0.2 — точнее.

**Решение:** композит. Формула выводится в БД как `SELECT score FROM v_problem_students`. Менеджер видит топ-20.

## ADR-12: Где хранить сессии — в БД, не в Redis

**Контекст:** в варианте со своим auth-слоем надо где-то хранить session_id → user_id.

**Решение:** таблица `user_sessions` в основной БД. Redis **не** разворачиваем на MVP.

**Почему:**
- Меньше одного сервиса в инфре.
- PostgreSQL managed всё равно уже есть — используем.
- Нагрузка низкая (100 юзеров) — Redis не даст выигрыша.
- V2: если начнёт тупить — добавим Redis как кэш сессий.

## ADR-13: Расписание — шаблон «день недели + время», не список событий

**Контекст:** у ученика регулярные уроки («пн/ср/пт 18:00»). Нужно не забыть провести урок и быстро его отмечать.

**Варианты:**
1. **Календарь событий** — таблица `lesson_events` с полным списком будущих уроков на ближайший месяц/год. Генерируется cron-ом.
2. **Шаблон** — таблица `student_schedules` (student_id, weekday, time). Повестка дня строится на лету: «на этот день недели найди слоты в расписании + уже записанные уроки».

**Решение:** вариант 2 — шаблон.

**Почему:**
- Меньше данных в БД, не нужен cron.
- Пропуск урока не плодит «осиротевшие» события.
- Смена расписания (с пн на вт) — одна правка, не пересоздание календаря.
- Повестка дня — один SELECT из SQL-функции `teacher_day_agenda(teacher_id, date)`.

**Что теряем:** сложные исключения («на 9 мая урока не будет») плохо ложатся. Если понадобится — V2 добавит `schedule_exceptions(student_id, date, skip | override_time)`.

## ADR-14: Перенос урока = одна запись с двумя датами

**Контекст:** «Ученик должен был прийти в понедельник по графику, а провели во вторник». Нужно отметить факт проведения, но не потерять, что урок был ПРОТИВ плана.

**Варианты:**
1. Две записи: «отменён в понедельник» + «дополнительный во вторник».
2. Одна запись с двумя полями дат: `scheduled_date = понедельник`, `lesson_date = вторник`.

**Решение:** вариант 2.

**Почему:**
- В отчётах виден и факт проведения (вторник), и плановое соответствие графику (понедельник).
- Учитель кликает одну кнопку «Перенос» вместо двух разных действий.
- Статистика «сколько уроков вне графика» — `where scheduled_date <> lesson_date`.
- Статистика «сколько проведено в понедельник» — `where lesson_date = ?`.
- Статистика «сколько запланировано на понедельник» — `where scheduled_date = ?`.

**Хранение:** колонки `lesson_date` / `lesson_time` — факт; `scheduled_date` / `scheduled_time` — план. Для уроков по графику они равны (бэкофилл в миграции).
